# Ducktape — Room 1 Multiplayer Prototype

Three separate browser clients (blind / mute / deaf), synced live over Firebase Realtime Database. Each duck plays on their own device.

## Files
- `index.html` — lobby: host or join a room, pick your role, start game
- `blind.html` — black background, white-outline-only room, walks into shapes placed by the deaf duck to hear tone cues, enters the 4-digit door code
- `mute.html` — sees the real randomized door code, places shape symbols in the room to signal digits (no text/voice)
- `deaf.html` — full color view of the room and the real duck model, sees symbols the mute duck places, no audio
- `shared/firebase.js` — Realtime Database read/write layer, room state shape documented at the bottom
- `shared/duck.js` — loads `assets/duck_baby.glb` if present, else falls back to a placeholder low-poly duck
- `assets/` — drop your converted duck GLB here (see assets/README.txt)

## Setup (5 minutes)
1. Go to https://console.firebase.google.com → create a project (free tier is fine)
2. In the project, go to **Build → Realtime Database → Create Database** (not Firestore)
3. Start in **test mode** for now (rules: `.read: true, .write: true` — tighten later)
4. Project settings → your web app → copy the config object
5. Paste it into `shared/firebase.js` in place of the `YOUR_...` placeholders
6. Serve the folder with any static server, e.g.:
   ```
   npx serve .
   ```
   (Can't just double-click index.html — ES module imports need http://, not file://)
7. Open the served URL on 3 devices/tabs, host a room on one, join with the other two, pick different roles, start once all three show ✅

## Swapping in your real duck model
Your uploaded `Duck_Baby.fbx` can't load directly in a browser — convert it once:
1. Blender → File → Import → FBX → select `Duck_Baby.fbx`
2. File → Export → glTF 2.0 (.glb) → save as `duck_baby.glb`
3. Drop it into `assets/`

All three clients pick it up automatically via `shared/duck.js`. If it's missing, the game still runs with a placeholder duck, so nothing breaks in the meantime.

## Current state of the "randomized keypad" logic
The door code is a random 4-digit string generated by `createRoom()` and stored at `/rooms/{code}/code`, visible only to the mute client. The shape→digit mapping in `blind.html` (`SHAPE_DIGIT_HINT`) is a placeholder stand-in for demo purposes — right now any shape always maps to the same digit regardless of room. For this to be a real puzzle, the mapping needs to be randomized per-room too and communicated through gameplay (e.g., the deaf duck sees a legend the mute duck doesn't, and has to convey "circle = 3" some other way) rather than hardcoded. That's the next design piece to solve before this becomes room-ready.

## Known gaps / what's still a stub
- Blind duck movement is keyboard-only (arrow keys/WASD) — needs touch controls for mobile
- No real puzzle yet forcing the deaf↔mute gesture relay to be interesting (it's currently a straight digit-to-shape lookup)
- No reconnect/rejoin handling if a player refreshes mid-game
- Firebase security rules are wide open (test mode) — must be locked down before any real deployment
- Only Room 1 exists; rooms 2–10 need their own asymmetric mechanic per your original brief
